# 课程版 v2：联机 / Rust 后端 / AI 玩家 / 规则改动 —— 开发流程与注意事项

> 本文是“工程落地流程”文档，不是玩法规则本身。  
> 玩法规则权威口径：
> - 重构版：`docs/game-rules-spec.md`
> - 课程版（本目录草案）：`docs/course-cn-v2/game-rules-spec.md`

## 1. 我们当前的约束与决策（基于讨论结果）

- 场景：课堂诚信/朋友局为主；预期同时在线约 100（房间数 * 每房人数）。
- 必需能力：断线重连、回放复盘（按 action log 重放）。
- 反作弊：初期不做强反作弊；隐藏信息仅做软隐藏（UI 不展示，不代表数据隔离）。
- 回合交互：顺序行动 + 响应窗口（不做 90 秒并行仲裁，降低实现与回放复杂度）。
- 内容扩展：职业/事件卡数据驱动，优先做到“加职业不改代码”（hooks/effects 白名单见 `docs/course-cn-v2/hooks-effects-schema.md`）。
- 社交：全局聊天/晒资产建议走联机事件日志（可回放），并做基础限流（见 `docs/course-cn-v2/multiplayer.md`）。
- （可选）救济与系统性风险：若启用“救济窗口/破产连带”，随机结果必须写入日志/回放（见 `docs/course-cn-v2/multiplayer.md`）。
- AI 玩家：全自动接管；开局前可为每个 AI 设置独立策略提示词；本局固定遵守；API Key 由房主配置并在房主端使用。
- 服务端权责（推荐基线）：不做“规则结算权威”，但做“联机权威”：
  - 权威排序：所有动作由服务端分配递增 `seq` 并持久化
  - 权威重连：基于 `snapshot + action log` 恢复
  - （建议）权威随机：骰子/抽牌结果由服务端生成并写入日志，避免分歧与“预知未来”

> 说明：如果未来真的需要“硬隐藏信息/防作弊/公开对战”，再升级为“服务端权威状态/结算”（见第 6 节）。

## 2. 推荐落地顺序（最少返工）

### Milestone 0：规则与引擎先定型（先文档、后代码）

交付物：
- 规则文档更新：`docs/game-rules-spec.md` 或 `docs/course-cn-v2/game-rules-spec.md`
- 引擎约束明确并可验证：确定性、可序列化、可回放（action log）

关键注意事项：
- 所有规则结算集中在 `lib/state/gameStore.ts`；组件只能触发 action/展示状态（P0 原则）。
- 任何需要“回放可复盘”的结算顺序都要进入日志（不要依赖 UI 的临时状态）。
- 禁止在规则里使用不可复现来源：`Date.now()`、未受控 `Math.random()`、依赖对象 key 遍历顺序等。

### Milestone 1：事件日志与重放（联机与回放的共同地基）

交付物：
- `Action` / `Event` 统一 schema（含版本号、幂等字段）
- `applyAction(state, action)` 可重复重放得到一致状态
- `snapshot`（全量状态快照）与 `rehydrate(snapshot, actions)`（从快照恢复）
- 状态 hash/校验（可选，但非常建议用于排查分歧）

DoD（完成标准）：
- 同一条 action log 在不同机器/不同刷新次数下重放后得到相同 `stateHash`
- 导出日志 → 清空本地 → 导入日志 → 能复盘到同一局面

### Milestone 2：联机协议定稿 + Rust 联机服务（房间/排序/持久化/重连）

交付物：
- WebSocket 协议与消息类型定稿（见 `docs/course-cn-v2/multiplayer.md`）
- Rust 服务端实现：房间、加入、广播、落库、重连、快照
- 基础权限：连接与 playerId 绑定；仅房主可代表 AI 玩家发 action

DoD：
- 断线重连：客户端重连后能继续同一局（状态一致）
- 回放：能从服务端导出某房间的 action log（可选带快照）并重放
- “重复发送”不产生重复提交（幂等）

### Milestone 3：客户端接入联机（不改变规则，只改 action 来源）

交付物：
- 前端把“本地 dispatch”改为“发送到 WS → 等服务端广播确认后再落地”
- 连接状态 UI（断线提示/重连按钮/房主标识）
- 冲突处理：非当前玩家（或非当前响应窗口参与者）的 action 直接拒绝（客户端 UI + 服务端最小状态机双层）
  - 说明：服务端不需要执行财务结算，但至少要能从已提交 action 推导出“当前轮到谁/是否处于响应窗口”，以便做时序校验与超时收敛；否则只能靠客户端拒绝，朋友局也能跑，但更容易卡局/分歧。

DoD：
- 两个浏览器加入同一房间，操作一方，另一方实时同步且不会出现不同步/跳回

### Milestone 4：AI 玩家（房主客户端托管）

交付物：
- 每个 AI 独立 `botConfig`：prompt、模型、超时、兜底策略；开局锁定
- AI 回合自动出招：调用 `/api/llm`（或直连 OpenAI）→ 产出 action → 校验 → 发送 WS
- AI 可见信息裁剪：统一用 `getPlayerView(state, playerId, visibility)` 生成 LLM 输入

DoD：
- AI 能在联机房间内完整跑完一局（含断线重连后继续）
- 回放不依赖再次调用 LLM（仅重放 action log）

## 3. 规则改动与联机/回放的耦合点（必须提前考虑）

- 一旦规则改动会影响 `Action` 类型/字段，必须：
  - 先更新对应规则文档
  - 再更新 action schema 与版本号
  - 明确旧日志是否还能回放：能 → 做向后兼容；不能 → 做迁移脚本或标记“旧局不可回放”
- “随机相关规则”（抽牌、洗牌、掷骰）是最容易导致客户端分歧的部分：
  - 推荐把结果作为事件写入日志（由服务端生成），而不是由客户端即兴产生

详见：`docs/course-cn-v2/rule-change-process.md`

## 4. Rust 重构的现实路径（避免一次性重写两套规则）

推荐两阶段：

1) 先 Rust 做“联机服务”（WS + 房间 + 持久化 + 重连 + 可选随机权威），不承载规则结算  
2) 等规则稳定、且确实需要硬隐藏/防作弊/公开对战，再考虑把规则引擎迁移到服务端（Rust or Node）

详见：`docs/course-cn-v2/rust-backend.md`

## 5. 手动验收清单（当前仓库无自动化测试时的最低保障）

- `npm run lint` 通过
- 单机：创建对局 → 掷骰/抽牌/结算 → 导出日志 → 重放一致
- 联机：两端加入同房 → 轮到谁谁能操作 → 断线重连继续 → 导出服务端日志回放一致
- AI：开局配置两名 AI 不同策略 → 房主托管 AI 自动行动 → 超时/失败有兜底，不会卡死回合

## 6. 何时需要升级为“服务端权威规则/状态”

如果出现以下任一目标，建议升级：
- 需要“硬隐藏信息”（其他客户端拿不到数据，而不是 UI 不显示）
- 需要防作弊/公开对战
- 需要服务端仲裁复杂窗口（例如全员响应窗口、异步拍卖）并保证可复盘

升级后的核心变化：
- 服务端持有完整 state，并按连接下发不同 `PlayerView`
- 客户端只发 action intent，服务端推进 state 并广播结果（或广播 patch）
