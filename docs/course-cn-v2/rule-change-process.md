# 规则改动流程（与联机/回放/AI 的耦合注意事项）

> 目标：让“改规则”不会把联机协议、回放复盘、AI 玩家一起带崩；把返工压到最低。

## 0. 先确认：你改的是哪一套规则？

- 重构版（当前主线）：先改 `docs/game-rules-spec.md`
- 课程版 v2（本目录草案）：先改 `docs/course-cn-v2/game-rules-spec.md`

无论哪套规则，工程实现的核心约束不变：
- 所有规则结算集中在 `lib/state/gameStore.ts`
- 组件只负责触发 action/展示状态
- 日志必须可复盘（同一回合内现金/现金流变化的顺序要可重放）

## 1. 规则改动“分级”——决定你需要动哪些地方

### A 级：纯数据改动（低风险）

例：卡牌数值、职业画像数值、文案/本地化；在白名单 hooks/effects 范围内新增/调整职业与事件卡数据。

- 通常只需要改 `lib/data/*` / `cardStrings.json` / i18n（若课程版 v2 仍处于“文档草案阶段”，也可能只改 `docs/course-cn-v2/*`）
- 对联机与回放影响很小（但仍建议 bump 数据版本，便于复盘定位）
- 职业/事件卡的数据结构与白名单口径见：`docs/course-cn-v2/hooks-effects-schema.md`

### B 级：结算逻辑改动（中风险）

例：收入/支出计算方式、回合结构调整、窗口/状态机新增。

- 必然要改 `lib/state/gameStore.ts`
- 必须检查：
  - 日志事件顺序是否仍可复盘
  - 是否引入新的“阻塞状态”（等待玩家输入/响应窗口）
  - 是否需要新的 `Action` 类型或字段
  - 是否新增/修改 hooks/effects 白名单（这会影响“加职业不改代码”的边界，需同步更新 `docs/course-cn-v2/hooks-effects-schema.md`）
  - 是否引入新的“随机判定”（例如破产连带随机选人）；若引入，建议把随机结果写入 action log（见 `docs/course-cn-v2/multiplayer.md`）

### C 级：动作协议改动（高风险，联机/回放强耦合）

例：新增/删除 action 类型；修改 action payload 的含义；改变随机来源（客户端→服务端）。

必须同时处理：
- 文档更新（规则 + 协议）
- action schema 版本号升级
- 旧局回放策略（兼容 / 迁移 / 明确不支持）

补充：即使“不影响结算”的联机事件（如聊天/晒资产），只要进入 action log，也属于协议层改动；建议同样走版本升级与兼容策略。

## 2. 标准改动流程（建议每次都按这个走）

1) 写清楚“改动点与边界条件”
   - 在规则文档中把“结算顺序/触发条件/例外情况”写成算法级描述
   - 把与旧版差异（避免复刻 bug）记录到审计文档（如有需要）

2) 列出受影响的状态字段与 action
   - state：哪些字段新增/删除/语义变化（影响快照与回放）
   - action：哪些类型新增/删除/字段变化（影响联机协议）

3) 先写/更新“可回放用例”（强烈建议）
   - 用一段 action 序列描述“期望最终状态”
   - 即使暂时没上自动化测试，也把用例写成可重复的 fixture（后续迁移 Rust/联机排查分歧时救命）

4) 实现引擎改动（只改 `lib/state/gameStore.ts` 与必要的类型/数据）
   - 规则逻辑不要散落到组件
   - 日志写入要覆盖关键结算（钱从哪来/到哪去）
   - 避免 `Date.now()` / 非受控 `Math.random()`（尤其是联机）

5) 更新联机/回放兼容策略
   - 如果 action schema 有变化：升级 `actionVersion`
   - 如果 state 结构有变化：升级 `snapshotVersion`
   - 明确旧版本日志如何处理：
     - 继续支持：做兼容分支（建议有限期）
     - 不支持：在导入/回放时给出明确错误提示

6) `npm run lint` + 手动验收
   - 单机回放一致
   - 联机重连一致（若联机已接入）
   - AI 不会因为新规则卡死（若 AI 已接入）

> 架构或跨模块数据流变更（例如新增后端服务、引擎拆分、协议改动）必须同步更新 `docs/architecture.md`。

## 3. 与 AI 玩家强相关的注意事项

- AI 不是“规则权威”，它只能产出 action；最终必须走同一套 action 校验逻辑
- LLM 输出必须强约束（JSON schema / function calling），并做本地校验与兜底
- 若你引入“隐藏信息”：
  - 对人类可以软隐藏（UI 不展示）
  - 但对 AI 的输入必须用 `getPlayerView(...)` 过滤（否则策略会被“喂到不该看到的信息”）

## 4. 常见坑（强烈建议提前规避）

- 把规则写进组件（联机/回放时必崩，且难排查）
- 让随机数发生在 reducer 内但不记日志（重放不一致）
- action 没有幂等字段（断线重发会造成重复买入/重复扣钱）
- 规则改了但没 bump 版本（旧日志回放出错却无法定位）
