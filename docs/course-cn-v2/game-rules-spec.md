# 家庭财务沙盘（课程版·中国化）统一规则草案（v0.1）

> 适用范围：用于把当前项目演进为“偏财商教学/课程化”的新玩法。  
> 设计目标：低复杂度、90 分钟可跑完、强复盘、内容与逻辑贴近现代中国现实，并尽量降低与既有同类 IP 的混淆风险（不沿用可识别术语、结构与卡面表达）。

版本口径（为避免歧义）：
- 本文只维护**一份规则**：草案 `v0.1`。
- 本文中标记为“可选模块”的内容属于同一份规则里的**模块开关**：房间创建时决定启用/关闭；未启用时相关动作/效果一律不可用。
- 本文中出现的“可选建议/可选配置”属于课堂节奏与难度选择：它不会改变动作集合，但可能改变频率/参数；建议同样写入房间元数据（或课程记录）便于复盘定位。
- 联机/回放时：房间元数据必须记录 `enabledModules`（见 `docs/course-cn-v2/multiplayer.md`），否则同一份 action log 可能因开关不同而复盘分歧。

建议的 `enabledModules` 取值（默认值）：
- `lifeCards`：启用第 4 章的“人生事件触发”（默认启用）
- `dueDiligenceQuery`：启用 9.1 的“尽调/征信查询”（信息摩擦动作）（默认关闭）
- `bailoutWindow`：启用 7.1 救济窗口（默认关闭）
- `bankruptcyContagion`：启用 7.3 破产连带（默认关闭）
- `playerLoans`：启用 9.5 玩家间借贷（默认关闭）
- `blackBoxProjects`：启用 9.7 黑盒项目/洗钱/审计与诉讼（常开玩法，默认启用）

## 0. 总体原则

1) **课程化优先**：流程稳定、结算统一、便于讲解与复盘；尽量减少“走格子触发”的随机性依赖。  
2) **低复杂度**：玩家每回合只做 1 次关键决策；玩家个人核心字段尽量控制在 6-10 个以内（项目/合同等对象属于互动扩展，不计入个人字段）。  
3) **统一结算**：不使用“走过发薪日才结算”的机制；每回合固定时间单位结束时统一结算现金流。  
4) **内容中国化**：职业/事件/资产/负债全部以现代中国日常可理解的场景表述（且为原创内容库）。  
5) **课堂诚信/朋友局优先**：初期默认不做强反作弊；“隐藏信息”仅做软隐藏（UI 不展示，不代表数据隔离）。若未来需要陌生人匹配/公开对战/硬隐藏信息，需升级为“服务端权威视图/状态”（见 `docs/course-cn-v2/dev-workflow.md`）。  
6) **数据驱动与可扩展**：职业被动能力与事件效果必须使用白名单 hooks/effects 的结构化表达，避免在引擎里为具体职业/卡牌写硬编码分支（见 `docs/course-cn-v2/hooks-effects-schema.md`）。  
7) **确定性与取整**：所有金额以“元”为单位存整数；涉及比例/倍率时按“四舍五入（0.5 绝对值向上）”取整；回放必须能复现同样结果。

## 1. 游戏概览

- 玩家人数：1-6（课堂分组建议 2-4 人/组）。  
- 单局时长：约 90 分钟。  
- 回合结构：建议 **8 回合（每回合=1 个季度）**。  
- 核心目标：
  - **达成财务自由（FI）**：满足规则定义的“被动收入覆盖基础支出 + 应急金达标”。  
  - **实现梦想**：FI 后支付梦想预算并满足梦想条件，率先达成者获胜。
- 结束与裁决：
  - 若任意玩家实现梦想：立刻结束对局并记录胜者。
  - 若第 8 回合结束仍无人实现梦想：默认按第 8 章“课堂评分模式”决出胜者（也可选择“无胜者，仅复盘”）。

> 注：课堂教学局建议把“评分模式”作为兜底裁决；若更偏竞赛，也可改成“全程积分赛”。

## 2. 组件与牌堆（建议最小集）

### 2.1 玩家面板（Scenario：职业画像）

每位玩家开局选择 1 张职业画像（全部为原创）：

- `salaryMonthly`：月薪（税后或等效到手，建议直接用到手，降低复杂度）。  
- `baseExpensesMonthly`：基础月支出（吃住行、基础赡养/教育等的合并项，开局固定）。  
- `startingCash`：起始现金。  
- `debtPaymentMonthly`：负债月供（可选；若开局有房贷/车贷/消费贷则计入）。  
- （建议）`healthMax`：健康上限（建议 `2`，用于“健康事件/手停口停”教学点；见 `docs/course-cn-v2/hooks-effects-schema.md`）。  
  - 开局默认 `health = healthMax`（不额外引入“起始健康”字段，保持低复杂度）。  
- （可选）`careerTier`：职业等级（0/1/2…，用于“进修/升职”）。
- （可选）`modifiers[]`：职业被动能力修正器（只能使用白名单 hooks；目的是“加职业不改代码”，见 `docs/course-cn-v2/hooks-effects-schema.md`）。

### 2.2 梦想项目（Dream：目标板）

每位玩家选择 1 张梦想项目（全部为原创）：

- `dreamCost`：一次性资金需求（从现金支付）。  
- `requirement`：1 条简单条件（例如“无高息负债/职业等级>=2/拥有某类资产>=1/应急金>=6个月”）。
  - 实现建议：`requirement` 复用 `Condition` 结构（见 `docs/course-cn-v2/hooks-effects-schema.md`），中文仅用于展示，禁止从文案解析规则。  
  - 口径说明：规则里的“应急金 N 个月”建议统一表达为 `cash >= baseExpensesMonthly * N`（例如 `N=6`），避免引入新的派生字段。  

### 2.3 牌堆

为控制复杂度，v0.1 建议只保留三类“抽取型”牌堆（合作项目不作为抽牌堆，见第 9 章）：

1) **宏观/政策牌（Global）**：每回合公开 `globalDraws` 张，对所有玩家生效。  
   - 默认：`globalDraws = max(1, ceil(playerCount / 3))`（人数越多，宏观事件越频繁；更贴近“人多市场更繁荣/更波动”）。  
   - 示例效果（都应为简单修正）：利率上调/下调、行业景气、房租/教育通胀、就业环境等。  
2) **机会牌（Opportunity）**：玩家选择“投资/副业”动作时抽 1 张并决策买/不买。  
   - 资产类型建议：指数基金/债基/REITs/小店合伙/技能变现副业等（可含“流动性”标签）。  
3) **人生事件牌（Life）**（可选触发，见第 4 章）：健康/家庭/工作变动/教育等，强调预算管理与风险意识。

## 3. 状态与结算口径（低复杂度版）

### 3.1 玩家核心字段

- `cash`：可用现金（默认唯一的流动资金；若启用 9.7 黑盒模块，则通常为 `visibleCash + hiddenCash`；若存在冻结限制，被冻结部分暂不计入）。  
- `salaryMonthly`：月薪（开局来自职业画像，可能被“升职/降薪”改变）。  
- `passiveIncomeMonthly`：被动收入（月）。  
- `baseExpensesMonthly`：基础支出（月）。  
- `debtPaymentMonthly`：负债月供（月）。  
- `netCashflowMonthly = salaryMonthly + passiveIncomeMonthly - baseExpensesMonthly - debtPaymentMonthly`

（建议最小补充，用于梦想条件/教学复盘）
- `health`：健康值（整数，范围 `0..healthMax`；默认开局为 `healthMax`）。  
- `healthMax`：健康上限（通常为 2）。  
  - 口径说明：v0.1 不强制“健康 = 收入倍率/行动惩罚”。健康对收入/行动的影响由职业画像与事件卡通过 hooks/effects 提供；若未配置相关修正，则健康仅作为事件触发与复盘字段。  
- `tags[]`：玩家标签（示例：`HighInterestDebt` 高息负债、`Defaulter` 被执行/失信等）。  
- `assetsByCategory`：资产类别计数（例如 `realEstate/stock/equity/other`），仅用于梦想条件与“晒资产”摘要；v0.1 不要求估值/买卖。  

### 3.2 每回合（季度）统一结算

- 季度净现金流：`netCashflowQuarter = netCashflowMonthly * 3`  
- 回合结算：`cash += netCashflowQuarter`

> 可选配置：若课堂希望更直观，也可以把回合定义为“每回合=1个月”，则不乘 3、总回合数改为 12 回合；建议把该配置写入房间元数据，避免回放定位困难。

### 3.3 统一口径：有效现金流与取整（强烈建议遵循）

为避免不同实现出现“同一局回放算出来不一样”，建议对以下口径做统一说明（偏产品表述，但足够落地）：

1) **现金口径（可用现金）**
   - 本规则正文中的 `cash` 一律指**可用现金**：未启用黑盒模块时等价于唯一现金；启用 `blackBoxProjects`（默认）后通常等价于 `visibleCash + hiddenCash`。  
   - `visibleCash` 与 `hiddenCash` 都可用于支付/转账；区别在于“法律后果与信息不对称”口径：默认起诉/执行只能强制执行 `visibleCash`，`hiddenCash` 不被民事诉讼执行。  
   - 冻结口径（若存在冻结效果）：被冻结的钱在冻结期内不可用于支付/转账，且不计入 `cash`（可用现金）；解冻后恢复计入。  

2) **有效月度口径（用于净现金流计算）**
   - `salaryMonthly / passiveIncomeMonthly / baseExpensesMonthly / debtPaymentMonthly` 是“基数”，通常只会被“升职/降薪、买入机会、还款降供”等动作修改。
   - 宏观牌/状态带来的“倍率/临时增量”属于“结算时修正项”，不建议直接改写基数，以便回放定位（哪些是基数变化，哪些是临时波动）。
   - 推荐的顺序（够直观，也够一致）：
     - 有效工资：先叠加所有工资倍率（多条倍率相乘），再按取整规则落到“元”。  
     - 有效基础支出：先叠加所有支出倍率（多条倍率相乘），再加上当期支出增量（如房租上涨 +500/月），最后按取整规则落到“元”。  
     - 有效月供：以基数为主，叠加当期月供增量（如利率上调/逾期惩罚），按取整规则落到“元”。
   - `netCashflowMonthly` 的计算使用上述“有效值”，避免出现“宏观牌写了工资 ×0.9，但只影响展示不影响结算”的口径漂移。

3) **持续回合（durationTurns）口径（用于宏观/状态类修正）**
   - `durationTurns=1` 建议口径：从产生当回合起生效，并影响**本回合**的回合末结算一次；回合结算完成后自动失效。
   - `durationTurns>1`：每经历一次回合末结算，剩余回合数 `-1`，直至归零失效。

4) **取整与边界**
   - 金额最终落地为“元整数”；涉及比例/倍率产生小数时，按“四舍五入（0.5 绝对值向上）”取整。
   - 建议约束：`baseExpensesMonthly >= 1`，`debtPaymentMonthly >= 0`；若数据出现 0 或异常值，评分与除法相关口径一律按 `max(1, baseExpensesMonthly)` 兜底，避免分歧与崩溃。

5) **按比例分配的取整（多人分摊时）**
   - 当一笔金额需要按比例分到多人（例如合伙项目现金流分配、本金返还、保证金补偿），先按比例算出每人的“理论份额”，并按取整规则落到“元”。
   - 若四舍五入后总和与应分配总额不一致：
     - 总和偏小：按本回合行动顺序（`turnOrder`）从前到后依次 `+1` 补齐差额，直到对齐。  
     - 总和偏大：按 `turnOrder` 从前到后依次 `-1` 扣减差额（跳过当期份额已为 0 的玩家），直到对齐。  
     这样既“守恒”，也保证可回放。

## 4. 回合流程（8 个季度）

记号：
- `turn`：当前回合编号（从 1 开始；默认 1..8；若采用月度制则为 1..12）。本文提到的 `turnGte/turnLte/lastUpdatedTurn` 等口径也按该编号理解。  
- `playerCount`：本回合仍参与行动与结算的玩家人数（默认等于开局人数）。

每回合按固定顺序执行：

1) **公开宏观牌**（全局事件）
   - 翻开 `globalDraws` 张 Global，对全体生效（只允许简单修正，如“本回合工资收入 ×0.9”或“房租类支出 +500/月”）。  
   - 默认：`globalDraws = max(1, ceil(playerCount / 3))`；若翻开多张，按翻牌顺序依次结算并写入日志（保证可复盘）。
   - v0.1 建议把宏观牌效果约束在“工资/支出/月供/标签”等可直接结算的白名单字段上（见 `docs/course-cn-v2/hooks-effects-schema.md`），避免引入“未建模的收益率/估值口径”导致实现分叉与回放分歧。示例可改写为：  
     - “本回合工资收入 ×0.9（durationTurns=1）”  
     - “房租/教育通胀：基础支出 +500/月（durationTurns=1）”

2) **玩家决策阶段（每位玩家本回合只能做 1 个动作）**
   - 按顺序依次行动（默认**每回合轮转起始玩家**：起始玩家索引每回合 `+1 (mod playerCount)`；写入日志便于复盘）。
     - 口径说明：本回合行动顺序（`turnOrder`）从“起始玩家”开始，按座位/玩家列表顺序依次循环；本回合内所有响应窗口（入股/救济/借贷回应等）默认沿用同一 `turnOrder`，并以它作为“同分/同现金”的确定性打破方式。
	   - 所有“需要其他玩家参与/表态”的规则，一律通过**响应窗口**处理：由被邀请/被允许的玩家按顺序依次响应；响应不消耗其本回合动作（用于控时与鼓励互动）。
	     - 工程落地建议：响应窗口应有超时与默认行为（推荐短硬超时：默认每位响应者 `20 秒`；轮到某玩家响应时单独计时，超时自动视为“拒绝/不参与”并写入日志），避免联机卡局。
   - 动作列表（建议，六选一）：
     - A. **投资/副业**：抽 1 张 Opportunity，选择买/不买；买入后通常改变 `passiveIncomeMonthly`（或产生一次性收益/亏损）。  
     - B. **职业成长**：支付进修成本（现金），提升 `salaryMonthly`（或提升 `careerTier`，用于梦想门槛）。  
     - C. **负债管理**：支付一笔现金以降低 `debtPaymentMonthly`（或清除“高息负债”标签）。  
     - D. **风险管理**：本回合不进行风险投资；可选择“建立安全垫”（例如提高应急金/降低杠杆/提前还款等）。  
       - 若启用可选模块 `dueDiligenceQuery`（见 9.1），查询动作建议也归入 D（消耗本回合动作 + 扣除尽调费用）。
	     - E. **项目与博弈（合伙/黑盒/审计/起诉/洗钱）**：进行“多人项目相关”的主动作（见第 9 章）。  
     - F. **尝试梦想**（仅稳定 FI 玩家）：按第 6 章结算；若成功则立刻胜利并结束对局。
   - 动作支付口径（建议明确，避免“隐形借钱”）：
     - 除“无条件汇款/救济/借贷”等明确的资金往来外，玩家主动选择的动作若需要立即支付现金/锁定本金，执行时必须满足 `cash >= cost`；现金不足则该动作不可执行。  
     - 系统结算造成的现金减少（事件/宏观牌/项目爆雷/投资亏损等）可以把现金打到负数；此类“资金缺口”在回合末进入第 7 章的资金紧张/破产处理。
   - 附加操作（保留“玩家间无条件汇款/赠与”机制）：
     - 在**自己回合**的决策阶段，玩家可额外执行一次无条件汇款（赠与）（不计入本回合 1 次动作）：对任意玩家进行 `cash.transfer`（v0.1 默认使用 `visible`；启用黑盒模块后也可按动作指定 `hidden`）。  
     - 汇款不创建债务/不要求还款；若希望建立“应还/实还记录”，请启用 9.5 借贷模块并以合同化方式记录。  
     - 为控时建议：每位玩家每回合最多执行 1 次无条件汇款；所有汇款必须写入日志/回放。  
     - 口径说明：7.1 救济窗口中的救济转账属于“响应窗口行为”，不计入此处的“自己回合无条件汇款次数”（但仍建议在救济窗口内做次数限制，见 7.1）。

3) **（可选模块：`lifeCards`）人生事件触发**
   - 为保留“人多事件多”的真实感，同时控时，建议采用：
     - 默认：每回合触发 `lifeDraws` 张 Life（依次结算并写日志）。  
       - `lifeDraws = max(1, ceil(playerCount / 2))`  
       - 选人顺序（确定性口径）：按“当前现金从低到高”排序（同现金按本回合行动顺序），取前 `lifeDraws` 名；写入日志避免复盘分歧。  
     - 口径说明：选人排序发生在“本回合所有玩家完成主动作及其相关响应窗口”之后；排序用的“当前现金”指此时刻的可用现金（`cash`，启用黑盒模块时包含 `visibleCash + hiddenCash`）。  
   - 若课堂节奏允许、希望更“波动”：可把 `lifeDraws` 提高到 `playerCount`（每回合每人 1 张）。  

4) **黑盒项目结算（`blackBoxProjects` 默认启用）**
   - 见第 9.7：对所有 `running` 项目结算当期利息/分红/到期还本，余额不足则标记 `collapsed` 并进入后续审计/起诉流程。

5) **回合末统一结算（季度结算）**
   - 对每位玩家执行第 3.2 的统一结算，并记录日志用于复盘。

6) **资金紧张/破产处理（始终执行；救济/连带为可选模块）**
   - 回合末统一结算后，若有人 `cash < 0`，系统进入“资金紧张处理”（见第 7 章）。
   - 口径说明：若本回合存在“到期关闭的合作项目”（见 9.2），其“关闭 + 返还本金/保证金”先结算完成，再进入资金紧张判定（避免把“应返还的钱”误算成缺口）。
   - 若同一回合有多人 `cash < 0`：在进入该阶段时一次性生成处理队列，按“当前现金从低到高（同现金按本回合行动顺序）”排序；队列锁定，不因后续救济/连带导致的现金变化而重排（避免回放分歧）。
   - 若启用 `bailoutWindow`：对每位资金紧张者依次开启一次救济响应窗口（7.1）。
   - 无论是否启用救济窗口：在该玩家处理结束后仍 `cash < 0`，视为发生一次“破产事件”，并应用 7.2 的后果（低保在本轮“资金紧张处理”的最后统一发放）。
   - 若启用 `bankruptcyContagion`：本回合最多触发一次连带判定（7.3）。
     - 口径说明：连带产生的 `victim` 不加入本轮资金紧张处理队列（无救济窗口），但会立即获得“视同破产”的状态后果，并在本轮结束时一并参与低保发放（见 7.3）。

7) **检查财务自由（FI）解锁**
   - 见第 5 章。

## 5. 财务自由（FI）判定（课程版建议口径）

为强调“稳定性 + 风险管理”，建议 FI 需要同时满足：

1) `passiveIncomeMonthly >= baseExpensesMonthly`（被动收入覆盖基础支出）  
2) `cash >= baseExpensesMonthly * 3`（应急金 ≥ 3 个月基础支出）  
3) （建议）连续满足 **2 个回合** 才算“稳定 FI”（避免一次性波动导致“伪 FI”）

实现建议（避免“解锁一次就永久”导致口径混乱）：
- `fiUnlocked` 视为**可变状态（每回合末重新计算）**：在“回合末统一结算 + 资金紧张处理”后按上述条件计算；满足则 `fiUnlocked = true`，不满足则 `fiUnlocked = false`。  
- `fiStreak`（建议新增字段）：连续满足 FI 条件的回合数；若本回合满足则 `+1`，否则归零；当 `fiStreak >= 2` 时视为“稳定 FI”。  
- 首次达成 FI / 首次达成稳定 FI 时建议写入日志，用于课堂复盘。
- 时点口径（避免“本回合内反复变更 FI 状态”）：`fiUnlocked/fiStreak` 只在“回合末统一结算 + 资金紧张处理”后更新，并在**下一回合**的决策阶段用于判断是否允许选择动作 F。

## 6. 梦想实现（胜利条件）

当玩家达到“稳定 FI”（建议口径：`fiStreak >= 2`）后，玩家在自己回合的“决策阶段”可以选择 **F. 尝试梦想**（占用本回合的 1 次动作）：
  - 口径说明：是否允许在本回合选择 F，以“上一回合末已达成稳定 FI”为准（见第 5 章的时点口径）。当回合的宏观冲击不会取消资格，但会影响你是否仍有足够现金支付 `dreamCost`。

1) 校验梦想 `requirement` 满足（使用 `Condition` 判定；禁止从中文文案解析）。  
2) 校验 `cash >= dreamCost`（这里的 `cash` 一律指可用现金；启用 9.7 黑盒模块时通常等价于 `visibleCash + hiddenCash`（冻结部分除外））。  
3) 从现金扣除 `dreamCost` 并立刻胜利结束对局。  

> v0.1 为保持低复杂度：不做“尝试失败后的退款/部分退款/分期支付”机制；若未来确有课堂需求，建议新增独立模块并明确数据结构与回放口径后再引入。

## 7. 失败/出局（课程版建议）

为避免课堂中途“玩家被淘汰影响教学”，建议默认 **不强制出局**，而是：

- 若 `cash < 0`：玩家进入“资金紧张”状态；若在本轮“资金紧张处理”（含救济窗口/低保等，若启用）结束后仍 `cash < 0`，视为发生一次“破产事件”（见 7.2），下一回合行动受限（只能选 C/D）；当回合末（含救济/低保处理后）`cash >= 0` 时视为“现金修复”，解除限制并恢复胜利资格。  

> 若用于竞赛玩法，可引入“信用额度/应急借款”但会增加复杂度；v0.1 暂不引入。

### 7.1 可选模块：救济窗口（Bailout Window）（推荐配合博弈使用）

> 目的：让“资金紧张”不只是个人问题，而能触发协商/救济/勒索等互动；同时给“连带破产”提供可控的提前干预点。

触发：
- 在“回合末统一结算”后，若某玩家 `cash < 0`，立刻为该玩家开启一次救济响应窗口。
  - 若同一回合有多名玩家现金为负：按第 4 章“资金紧张/破产处理”的顺序逐个开启窗口（一次只处理一人，避免并发窗口导致回放分歧）。

流程（顺序行动 + 响应窗口）：
1) 系统广播：某玩家现金为负、缺口金额 `shortfall = max(0, -cash)`。  
   - 例：`cash = -12000` 则 `shortfall = 12000`；`cash = 500` 则 `shortfall = 0`。  
2) 其他玩家按本回合行动顺序依次选择是否救济：对该玩家进行 `cash.transfer`（救济转账不消耗出资者本回合动作）。  
   - 为控时建议：每位救济者在**同一个救济窗口**内最多救济 1 次（金额自定）；超时未响应视为“拒绝/不参与”。  
3) 窗口结束后再检查该玩家 `cash`：
   - 若 `cash >= 0`：危机解除，不触发后续惩罚/连带。  
   - 若仍 `cash < 0`：视为发生一次“破产事件”（记录日志并应用 7.2 的限制/资格规则）；若启用“破产连带”，再进入 7.3 的判定；本轮“资金紧张处理”结束时按 7.2 统一发放低保。

说明：
- 救济是否有利息、是否形成借贷合同，建议交给玩家自行协商（可选用 9.5 借贷模块进行“明示合同化”）。
- 课堂诚信/朋友局不做强反作弊，转账属于公开可复盘事件，建议写入日志用于复盘。

### 7.2 破产后果（v0.1 定案）

触发：
- 在本轮“资金紧张处理”中，对某玩家完成救济窗口（若启用）后，若该玩家仍 `cash < 0`，视为发生一次“破产事件”。  
- 未启用救济窗口时：在本轮“资金紧张处理”中，以“回合末统一结算”后的 `cash < 0` 直接判定为“破产事件”。

后果（规则）：
- **不强制淘汰**：破产者可继续参与后续回合与讨论（课堂教学优先）。  
- **胜利资格暂停（可恢复）**：当玩家处于 `cash < 0` 的“资金紧张/破产”状态时，暂停其胜利资格；当其在回合末（含救济/低保处理后）`cash >= 0` 时，视为“现金修复”，胜利资格自动恢复。  
- **行动限制**：从下一回合起，该玩家在“玩家决策阶段”只能选择 **C. 负债管理** 或 **D. 风险管理**，直到其回合末（含救济/低保处理后）`cash >= 0`。  
  - 口径说明：行动限制只限制其在自己回合“主动作”的选择，不限制其在他人的响应窗口中做出“同意/拒绝”等表态；但凡涉及出资/转账/支付，仍需满足现金足够的动作支付口径（见第 4 章）。
- **低保救济（小额）**：为保证参与感，当发生破产事件后，在本轮“资金紧张处理”的最后，系统为所有仍 `cash < 0` 的玩家发放一次低保：  
  - （若启用 7.3 破产连带：低保在连带判定完成后统一执行）  
  - `shortfall = max(0, -cash)`（以低保发放前为准）  
  - `welfareCap = max(1000, round(baseExpensesMonthly * 0.5))`  
  - `welfare = min(shortfall, welfareCap)`  
  - `cash += welfare`（低保不使现金变为正数；最多补到 `cash = 0`）  
- **不做额外自动修复**：除上述低保外，不做“归零/免债/强制借款”等自动修复，避免遮蔽决策后果。  
- **必须可复盘**：破产事件与触发时点必须写入日志/回放（至少包含 `playerId/cash/shortfall`）。

### 7.3 可选模块：破产连带（随机牵连一人）（系统性风险）

> 目的：提供“威胁手段”，让其他玩家在面对资金紧张者时有动力救济；同时引入现实中的“系统性风险/黑天鹅”讨论点。  
> 注意：该规则随机性强、情绪冲击大，建议教师明确告知并控制频率。

触发：
- 当发生一次“破产事件”（见 7.2）后，可进行一次“破产连带”判定。
  - 为控时与避免“多次随机”导致课堂节奏失控：默认只对**本回合首次发生**的破产事件做连带判定；若本回合后续还有其他破产事件，不再触发连带。

判定：
- 以概率 `contagionChance` 触发连带（建议默认 `50%`，可按课堂风格调整）。  
- 若触发：从“除破产者外的任意玩家”中随机选择 1 名作为 `victimPlayerId`。

效果（建议口径，保持简单）：
- 设破产者缺口 `shortfall = max(0, -bankrupt.cash)`（以救济窗口结束时、低保发放前为准）。  
- 触发“连带破产”（强制使被牵连者现金为负，避免“victim 资金充足就无所谓”）：
  - `victim.cash = min(victim.cash - shortfall, -shortfall)`  
  - 等价实现：先执行 `victim.cash -= shortfall`；若结果仍 `>= 0`，则置为 `victim.cash = -shortfall`。
  - 破产者现金不自动修复，仍保持 `cash < 0`（破产者需要通过下一回合的自身经营/再度获得救济来脱困）

victim 破产待遇（规则）：
- **视同破产**：`victim.cash` 被强制置为负后，`victim` 视同发生破产，适用 7.2 的“破产后果”（胜利资格暂停/可恢复 + 下回合行动限制 + 低保等）。  
- **无救济窗口**：`victim` 不会因为这次连带而在同一轮“资金紧张处理”中再次开启新的救济响应窗口；其后续救济处理留到 `victim` 自己下一回合末的“资金紧张处理”再执行（但本轮仍会按 7.2 获得低保）。  
- **不允许进一步连带（传播深度=1）**：`victim` 不会因为这次被连带而触发新的“破产连带”，避免“一人破产导致全员连锁破产”。  

约束（规则）
- 每回合最多触发 1 次连带（避免链式爆炸拖慢课堂节奏）。  
- 单人局或只剩 1 名玩家时不触发连带。  
- 连带的随机结果（是否触发、被选中的 victim）必须写入日志/回放，避免复盘分歧（联机建议写成权威随机事件，见 `docs/course-cn-v2/multiplayer.md`）。

## 8. 课堂评分模式（8 回合结束兜底，推荐）

若第 8 回合结束仍无人实现梦想，默认在 90 分钟结束时按积分评估（也可选择“无胜者，仅复盘”）：

- FI 达成（推荐默认值）：  
  - `fiUnlocked=true`：+10 分  
  - `fiStreak>=2`（稳定 FI）：额外 +5 分  
- 梦想进度（推荐默认值，避免引入“分期支付/退款”）：  
  - 梦想资金准备度：`dreamReadyRatio = min(1, max(0, cash / dreamCost))`，得分 `round(10 * dreamReadyRatio)`（这里的 `cash` 一律指可用现金，见第 3.3）  
  - 梦想条件满足：若 `requirement` 已满足，额外 +5 分  
- 风险控制（推荐默认值）：  
  - 应急金月数：`emergencyFundMonths = max(0, floor(cash / max(1, baseExpensesMonthly)))`，得分 `min(6, emergencyFundMonths)`（0..6 分；这里的 `cash` 一律指可用现金，见第 3.3）  
  - 负债压力：`debtRatio = debtPaymentMonthly / max(1, salaryMonthly + passiveIncomeMonthly)`，若 `debtRatio=0` +4 分；`<=0.25` +3 分；`<=0.5` +1 分；否则 +0 分
    - 口径说明：评分模式的 `debtRatio` 分母使用“收入基数”（`salaryMonthly + passiveIncomeMonthly`），不随当回合宏观倍率波动；便于课堂对比“结构性负担”。

并列裁决（建议口径，避免课堂争议）：
- 若总分相同：先比较 `dreamReadyRatio` 更高者；若仍相同，再比较 `cash` 更高者；若仍相同，按本回合行动顺序 `turnOrder` 更靠前者胜（或并列胜利，由课堂决定）。

该模式能避免“有人提前通关导致其余玩家无聊”，也更适合教学复盘。

## 9. 玩家间合作与博弈（共赢 + 允许背叛 + 社交沟通）

本章为课程版新增的“互动系统”机制：让玩家体验协作、契约、信任与风险的权衡，并通过沟通/信息披露形成博弈空间。规则设计目标是**鼓励合作创造增量**，同时允许在明确代价下进行“违约/背叛”。

### 9.1 信任与公开履约记录（无系统信誉值）（v0.1 定案）

为更贴近现实并增加博弈空间，课程版 v2 **不引入**“数值信誉（Reputation）”与其带来的系统强制限制（例如信誉为 0 就禁止合作/借贷）。  

合作是否发生、是否继续合作、是否愿意借钱/投资，**完全由玩家自行判断**；系统只提供“可复盘的公开信息”作为判断依据。

建议提供的公开信息（不改变结算，仅用于决策与复盘）：
- **履约记录摘要**：完成/违约的合作项目数量与金额、借贷是否按期偿还、是否发生过起诉与执行。  
- **资金与资产的自愿披露**：玩家可通过“晒资产/晒摘要”在公频建立信任（见 9.6）。  
- **法律/监管标签（可选）**：如启用起诉/执行模块，被执行后可标记 `Defaulter`（公开可见）。该标签用于教学讨论“法律后果/信用污点”，但不替代玩家的主观判断。

信息摩擦（建议明确，增加博弈空间）
- **滞后性**：履约记录摘要建议以“回合末结算后的快照”为准，并标记 `lastUpdatedTurn`。当回合内发生的违约/还款/挪用等行为，默认要到**下一回合**才会出现在系统摘要里（除非当事人自行在公频披露或被他人当场发现）。  
- **查询成本**（可选模块：`dueDiligenceQuery`，但推荐用于增加博弈空间）：若玩家希望查看某一玩家的“更详细履约记录”（例如最近 N 次合作/借贷的条目明细），建议必须在自己回合选择 **D. 风险管理** 并支付“征信/尽调费用”，系统再以**私信**返回查询结果。  
  - 费用建议：`ddCost = max(500, round(baseExpensesMonthly * 0.1))`（可按课程难度调整）。  
  - 查询频率建议：每位玩家每回合最多查询 1 次（本质上已被“每回合 1 个动作”限制）。  
  - 查询结果建议同样基于 `lastUpdatedTurn` 的快照，避免“实时看穿”导致信息不对称坍塌。  
  - 该机制不等价于系统信誉门槛：查询到信息后如何解读、是否继续合作，仍由玩家自行判断。
- **污染口径（建议明确）**：公开履约记录摘要只应包含“系统可验证的事实事件”（如：项目按约完成/违约、项目爆雷/无法支付、借贷到期未还、起诉/执行完成等）。  
  - 玩家在公频的指控/自述默认不自动写入履约记录（因为可能是谣言、误导或策略性话术）；它们只作为社交层信息，由玩家自行判断可信度。

### 9.2 合作项目（合伙 Venture）

合作项目是一类“多人凑资、按份额共享现金流”的机会，用于鼓励玩家协作完成单人难以完成的目标。

- 触发：玩家在决策阶段选择 **E. 项目与博弈**，并选择“合作项目（合伙 Venture）”。  
- 项目卡（Venture）建议字段（后续以数据驱动实现）：
  - `capitalRequired`：项目所需总本金（必须凑齐才成立）。  
  - `durationTurns`：持续回合数（建议 `1..3`）。  
  - `cashflowMonthlyTotal`：项目总月现金流（项目进行中时生效）。  
  - `depositRate`：发起人保证金比例（建议固定为 `20%`）。  
- 发起与入股（响应不占用其他玩家本回合动作，以鼓励合作）：
  1) 发起人公布项目，并声明自己出资额（可为 0 但不建议）。  
  2) 其他玩家按顺序选择是否入股以及出资额。  
  3) 每位玩家每回合最多参与 **1 个** 合作项目（发起或入股都算），以控制时长与节奏。  
  4) 若总出资达到 `capitalRequired`：项目成立并进入“进行中”；否则所有出资原路退回，项目作废。  
  - 口径说明：项目成立时，发起人需要同时支付“自己承诺的出资 + 保证金”（见下文）；任一方现金不足则视为无法出资/无法发起（避免形成未建模的信用垫付）。
- 资金锁定与收益：
  - 项目成立后，所有出资记为“锁定本金”，在项目结束前不可动用。  
  - 发起人额外支付保证金：`deposit = round(capitalRequired * depositRate)`（锁定）。  
  - 项目进行中：按出资比例把项目现金流计入被动收入：  
    - `参与者 passiveIncomeMonthly += cashflowMonthlyTotal * (个人出资 / 总出资)`  
    - 口径说明：“总出资”指凑齐 `capitalRequired` 的出资总额（不含保证金）。按比例分配与取整遵循第 3.3 的“按比例分配取整”口径，保证总和守恒。  
  - 到期：项目关闭，撤销上述被动收入增量；锁定本金按出资比例返还；发起人保证金返还。  
- 结算时点（建议明确，保证可回放且便于讲解）：
  - 项目成立后立即进入 `running`：从**本回合**的“回合末统一结算”开始计入现金流（即会影响本回合的季度净现金流）。  
  - `durationTurns` 以“回合末结算次数”为单位：项目每经历一次回合末结算，剩余回合数 `-1`；当剩余回合数归零时，在**本回合结算完成后**执行“项目关闭 + 返还本金/保证金”（返还发生在资金紧张处理与 FI 检查之前）。  
- 协同加成（鼓励共赢，可选但推荐）：  
  - 若参与者人数 `>= 2`，则 `cashflowMonthlyTotal` 额外 `+10%`（四舍五入），表示协作带来的效率/资源加成。  

### 9.3 允许背叛：发起人违约/挪用（可选规则，推荐开启以增加博弈）

为引入博弈性，允许发起人在承担明显代价的前提下选择“违约/挪用”：

- 时点：发起人在自己决策阶段可选择“违约/挪用”（占用本回合动作）。  
- 结算口径（固定、低复杂度）：
  1) 项目立即终止：不再产生现金流；撤销项目带来的被动收入增量。  
  2) 锁定本金处理：  
     - 发起人获得一次性现金：`embezzlement = round(锁定本金总额 * 0.15)`  
     - 剩余本金：`locked - embezzlement` 按出资比例返还给**所有**参与者（含发起人）。  
     - 发起人保证金 **全额没收**，按出资比例分配给“除发起人外”的参与者作为补偿。  
     - 口径说明：保证金补偿的“出资比例”以“除发起人外参与者的出资额”为分母（不包含发起人的出资），避免补偿被发起人自己分走。  
     - 口径说明：涉及“按比例返还/分配”的环节均遵循第 3.3 的“按比例分配取整”口径，保证总额守恒且可回放。  
  3) 该违约行为应被记录进“公开履约记录”（见 9.1）；后续是否还能获得投资/借款，完全由玩家基于信息判断。  

> 说明：将 `embezzlementRate` 设为小于 `depositRate`（默认 15% < 20%）可以保证“违约/挪用”存在**即时、可见的财务代价**，避免其在数值上成为支配策略；但玩家仍可能为了“短期现金/博弈威胁/阻止项目亏损扩大”等原因选择违约，保留教学讨论空间。
>
> 口径澄清（与 9.7 区分）：本节的“违约/挪用”是**公开、可验证**的行为，挪用所得默认进入 `visibleCash`（可见钱包；属于可用现金的一部分）。  
> 在 9.7 黑盒模块下（默认启用），玩家可以在**后续回合**再选择“洗钱”把一部分 `visibleCash` 转为 `hiddenCash`（额外动作 + 手续费/折扣 + 风险）；默认不做“挪用即自动洗钱”，避免把两种教学点（背叛 vs 合规/信息不对称）混在一次动作里。

### 9.4 信任修复（非系统数值，完全由玩家判断）

本规则不提供“一键洗白/花钱加信誉”的系统通道。若玩家违约或被质疑：
- 只能通过**实际履约**（按期分红/还款、主动补偿、提供更高保证金/抵押等）与**信息披露**（晒摘要/公开承诺）来重新建立信任；  
- 是否接受修复、是否继续合作，由其他玩家自行决定。

### 9.5 可选模块：玩家间借贷（信用合作 + 允许拖欠）

若课程希望覆盖“信用/契约/违约”主题，可开启玩家间借贷模块（建议教师先讲清楚再启用）：

- 借贷为双方自愿：出借人给出借款本金，借款人承诺在未来回合归还（可含利息）。  
- **借贷 ≠ 汇款**：  
  - “汇款/赠与/救济”是**无还款义务**的转账（例如救济窗口里的救济）。  
  - “借贷合同”是在转账基础上创建**可复盘的债务约定**，用于教学讨论“信用/违约/追偿”。  
- 合同字段（建议最小集，避免实现爆炸）：  
  - `principal`：本金（元，整数）  
  - `dueInTurns`：到期回合差（必须为有限正整数；建议 `1..3`，课堂可调）  
  - `interestRatePerTurn`：每回合利率（允许为 `0`，不允许为负；例如 `0.05` 表示 `5%`；本局每回合默认=1 个季度，因此该利率是“每回合口径”）  
  - 还款方式（v0.1 推荐口径）：到期一次性归还 `principal + principal * interestRatePerTurn * dueInTurns`（按“确定性与取整”规则取整）  
- **不允许“零利率 + 无限期还款”的合同**：这在效果上等价于赠与，会让履约记录与教学讨论失去意义。若双方希望无偿支持，请直接做“汇款/赠与/救济”（或在救济窗口中执行）。  
- 允许拖欠/违约：借款人到期可选择不还（或只还部分）。是否继续借、是否公开曝光，由出借人与其他玩家自行判断。v0.1 不提供“借贷起诉/强制执行”规则（9.7 的起诉/执行仅用于黑盒项目，避免复杂度爆炸）。  
- 建议将“借贷合同 + 应还/实还记录”纳入公开履约记录摘要（见 9.1），用于复盘与教学讨论。

### 9.6 社交与信息系统（公频聊天 + 晒资产）（推荐）

> 课堂诚信/朋友局推荐开启：它不改变结算规则，但显著增强“信任/舆论/协商”的教学讨论空间。  
> 工程上建议把聊天事件也写入 event log（便于回放复盘），但聊天本身不应影响数值结算。

- **全局公频（Global Chat）**：全程开启，允许发送文本与表情。  
  - 用途示例：项目宣传、风险预警、舆论博弈、协商分工。  
- **防刷屏**：建议服务端按连接做基础限流（例如 `1 秒 1 条`，超出拒绝/丢弃并提示）。  
- **晒资产（Show Off）**：玩家可主动把“可公开摘要”推送到公频，持续 `10 秒`（UI 层展示即可）。  
  - 课堂诚信/朋友局默认仅软隐藏：晒不晒由玩家自主；即使不晒，技术上仍可能被看到（不做强反作弊）。  
  - 可公开摘要建议只包含：职业、当前被动收入/基础支出区间、公开资产数量（或资产类别计数）、履约记录摘要（完成/违约/被起诉次数）等，避免直接暴露全部细项。

### 9.7 常开模块：黑盒项目 / 洗钱 / 审计与诉讼（`blackBoxProjects`）

> 本模块是课程版 v2 的常开核心互动：信息不对称 + 追偿成本 + 社交博弈。  
> 若课堂确实需要降低复杂度，可在建房时关闭 `blackBoxProjects`，退回到“单钱包 + 无诉讼”的轻量版（但会损失该模块的教学点）。

#### 9.7.1 规则参数（推荐默认值，建议写入房间元数据）

- `launderRate = 0.8`（洗钱到账比例；默认等价于 20% 手续费/折扣）  
- `auditFee = max(2000, round(baseExpensesMonthly * 0.5))`  
- `lawsuitFee = max(5000, round(baseExpensesMonthly * 1.0))`  
  - 口径说明：上述费用以“支付者自身的 `baseExpensesMonthly`”为基数计算；起诉加入者若选择加入，同样按自己的基数支付 `lawsuitFee`。  
- 条款菜单（建议 v0.1 固定为离散档位，避免玩家自填导致失衡与争议）：
  - `targetCapitalOptions = [20000, 40000, 60000]`  
  - `promisedRateOptions = [0.05, 0.10, 0.15]`（每回合固定利息比例）  
  - `durationTurnsOptions = [1, 2, 3]`  
- 控节奏建议：
  - `maxBlackBoxParticipationPerTurn = 1`（每位玩家每回合最多参与 1 个黑盒项目：发起或投资）  
  - `maxRunningBlackBoxProjectsPerPlayer = 1`（每位玩家同时最多 1 个进行中的黑盒项目）  
  - `maxLawsuitPerProject = 1`（每个项目最多清算一次）

#### 9.7.2 现金拆分（与第 3 章口径一致）

- `visibleCash`：可用于支付/转账；在起诉/执行时可被强制执行  
- `hiddenCash`：洗钱后获得；同样可用于支付/转账；默认不可被民事诉讼执行（冻结口径见第 3.3）  
- 未启用本模块：可视为只有一个现金口径（`cash`），等价于 `visibleCash`，且 `hiddenCash=0`。

#### 9.7.3 核心对象：黑盒项目（Project）

项目为独立对象，建议至少包含：
- `projectId / ownerPlayerId / name`
- `targetCapital`：募集目标（来自条款菜单）  
- `promisedRate`：承诺利息比例（每回合口径，来自条款菜单）  
- `remainingTurns`：剩余回合数（来自条款菜单）  
- `status`：`fundraising | running | collapsed | closed`
- `publicLiability`：公开可见的“尚未兑付的本金总额”（默认等于投资本金总额；兑付本金/清算赔付后减少）  
- `realBalance`：项目真实余额（仅项目方可见；课堂诚信局为软隐藏）  
- `investors[]`：投资人列表（至少包含 `playerId/principal`）

公开信息口径：
- `targetCapital/promisedRate/remainingTurns/publicLiability/investors/status` 对所有玩家可见，用于信任博弈与复盘。  
- `realBalance` 默认仅项目方可见；审计后对审计人可见（见 9.7.7）。

#### 9.7.4 发起募资（主动作，建议归入第 4 章动作 E）

约束：
- 项目方同时只能拥有 1 个 `running` 的黑盒项目（建议默认）。  

流程（顺序行动 + 响应窗口）：
1) 项目方在自己回合选择“发起黑盒项目募资”（占用本回合 1 次主动作），并从条款菜单选择 `targetCapital/promisedRate/remainingTurns`。  
2) 系统开启募资响应窗口：其他玩家按本回合行动顺序依次选择是否投资与投资金额（投资不消耗投资人的主动作；超时视为拒绝/不参与）。  
   - 口径说明：投资一旦确认，资金立即从投资人的 `visibleCash` 扣除并计入项目的 `realBalance`；同时在 `investors[]` 中记录其 `principal`，并将 `publicLiability` 增加同额（表示“已收本金但尚未兑付的承诺”）。投资人在募资期内视为“本金已锁定”，不能再动用这笔钱。  
3) 募资结束时判定是否成立：  
   - v0.1 定案：**必须筹够 `targetCapital` 才能成立**。  
   - 若募集到的本金总额 `>= targetCapital`：项目进入 `running`（保持上述已入账的 `realBalance/publicLiability/investors[]`），并从下一次“项目结算”开始按约兑付。  
   - 若 `< targetCapital`：项目作废，所有投资本金原路退回（从 `realBalance` 退回到各投资人的 `visibleCash`），并回滚该项目的 `publicLiability/investors[]` 变化；项目不进入 `running`。  

控节奏建议（默认）：
- 每位玩家每回合最多参与 1 个黑盒项目（发起或投资），避免同回合多窗口导致失控。

#### 9.7.5 每回合项目结算（第 4 章步骤 4）

对所有 `running` 项目依次结算（顺序建议按项目创建时间或 `projectId` 排序，保证可回放）：

1) 计算当期应付利息（固定、低复杂度）：
   - `totalPrincipal = sum(investor.principal)`  
   - `interestDueTotal = round(totalPrincipal * promisedRate)`  
   - 利息分配：按投资本金比例把 `interestDueTotal` 分给投资人，取整与守恒遵循第 3.3 的“按比例分配取整”。  
2) 支付与爆雷判定：
   - 若 `realBalance < interestDueTotal`：项目立刻进入 `collapsed`（v0.1 定案：当期不做部分支付），并停止后续兑付；保留 `realBalance` 作为后续审计/起诉的可执行范围之一。  
     - 口径说明：进入 `collapsed` 后，不再自动支付任何利息/本金；`realBalance` 不自动按比例分配给投资人，只能通过后续审计/起诉（或玩家私下协商）解决。  
   - 若 `realBalance >= interestDueTotal`：从 `realBalance` 支付利息给各投资人（建议利息进入投资人的 `visibleCash`），然后 `realBalance -= interestDueTotal`，`remainingTurns -= 1`。  
3) 到期还本：
   - 若 `remainingTurns` 归零：项目需一次性归还全部本金 `totalPrincipal`（按每位投资人的 `principal` 原路返还，建议进入投资人的 `visibleCash`），并将 `publicLiability` 减至 0。  
   - 若到期时 `realBalance < totalPrincipal`：视为“到期爆雷”，项目进入 `collapsed`（v0.1 定案：不做部分还本），保留 `realBalance` 供后续追偿。  
   - 若还本成功：`realBalance -= totalPrincipal`，项目进入 `closed`。

#### 9.7.6 挪用与补资（主动作，建议归入动作 E）

- **挪用**：仅项目方可在自己回合执行（占用本回合主动作）：把 `realBalance` 的一部分转入项目方的 `visibleCash`。  
  - 口径建议：挪用本身不直接进入公开履约记录（避免信息不对称坍塌）；更适合在“爆雷/起诉执行完成”等可验证事件发生后更新公开负面记录。  
- **补资**：仅项目方可在自己回合执行（占用本回合主动作）：把自己 `visibleCash` 的一部分转入项目 `realBalance`，用于避免爆雷或争取到期兑付。

#### 9.7.7 洗钱（主动作，建议归入动作 E）

- 把 `visibleCash` 的 `X` 转为 `hiddenCash` 的 `round(X * launderRate)`：  
  - 约束：执行时必须满足 `visibleCash >= X`。  
  - 默认推荐：不做“挪用即自动洗钱”，保留成本/风险/可见性的选择空间。

#### 9.7.8 审计（主动作，建议归入动作 E）

- 任何玩家可对某项目发起审计（占用本回合主动作 + 支付 `auditFee`）。  
- 审计结果默认私密发送给审计人，建议至少包含：  
  - `realBalance`（当前真实余额）  
  - `totalPrincipal`（募集到的本金总额）  
  - 是否足以覆盖下一次 `interestDueTotal` / 是否足以到期还本  
  - （可选）累计挪用金额与最后挪用回合（若实现了相应统计）

#### 9.7.9 起诉与执行（主动作，仅 `collapsed` 可用）

触发：
- 只有当项目 `status=collapsed`（含到期爆雷），投资人才能对该项目发起起诉。

流程（发起 + 加入窗口 + 清算）：
1) 任一投资人在自己回合选择“起诉”（占用本回合主动作），并支付 `lawsuitFee`。  
2) 系统开启“加入起诉”响应窗口：该项目的其他投资人按本回合行动顺序选择是否加入（加入者同样需支付 `lawsuitFee`；超时视为不加入）。  
3) 窗口结束后立即执行一次性清算（每项目最多一次）：  
   - 可执行范围：`project.realBalance + owner.visibleCash`（不执行 `owner.hiddenCash`）。  
   - 赔付范围（v0.1 定案）：仅对“参与起诉的投资人”按其本金比例分配赔付；最多赔到各自本金为止；**不追利息**（包括当期与历史未付利息，避免规则膨胀）。  
     - 计算口径（建议明确，避免实现分歧）：
       - `plaintiffsPrincipal = sum(plaintiff.principal)`  
       - `recoverable = project.realBalance + owner.visibleCash`  
       - `payoutTotal = min(recoverable, plaintiffsPrincipal)`  
       - 将 `payoutTotal` 按各原告 `principal` 比例分配给原告，取整守恒遵循第 3.3 的“按比例分配取整”；赔付建议进入原告的 `visibleCash`。  
       - 扣款顺序建议：优先从 `project.realBalance` 扣到 0，不足部分再从 `owner.visibleCash` 扣（保证“项目资金先清算”更符合直觉，也更利于复盘讲解）。  
   - 若仍有剩余可执行金额（因为部分投资人未起诉或本金已赔足）：剩余返还给项目方（进入 `owner.visibleCash`）。  
   - 清算完成后：项目进入 `closed`，并将 `publicLiability` 减少 `payoutTotal`（剩余部分可视为坏账，用于复盘摘要）。  
4) 起诉/执行完成后：项目方添加 `Defaulter` 标记（公开可见），并写入公开履约记录摘要（默认滞后到下一回合更新，见 9.1）。

## 10. 内容与表达约束（避免混淆风险的产品约束）

1) 不使用可识别的旧术语作为阶段名/按钮/文案（例如特定双阶段命名、特定发薪日触发结构）。  
2) 职业、事件、资产、梦想的名称与文本全部原创；避免“整句照搬/同义改写”。  
3) 结构上优先采用“时间轴回合制（季度/月度）统一结算”，而非“双环绕圈发薪”的体验。
4) 任何会影响结算/可回放的一致性逻辑，必须以结构化字段表达；禁止从中文文案解析规则（hooks/effects 口径见 `docs/course-cn-v2/hooks-effects-schema.md`）。

## 11. 数据驱动扩展约定（工程对齐）

> 本节用于支持“内容中国化 + 可扩展”，并落实“加职业不改代码”的约束。

- 职业：建议由 `profile + modifiers[]` 组成；`modifiers[]` 只能使用白名单 hooks（见 `docs/course-cn-v2/hooks-effects-schema.md`）。
